# 刷题答案第二卷：操作系统、网络与图形学

## 操作系统与并发

### 96. 进程、线程与协程

进程是资源隔离和保护的基本单位，拥有独立虚拟地址空间；线程是 CPU 调度的常见单位，同进程线程共享地址空间和大部分资源；协程通常由用户态运行时协作调度，切换开销低，但阻塞系统调用仍可能阻塞承载线程。

### 97. 用户态与内核态

应用通过系统调用、异常或中断进入内核态，CPU 切换权限级别并跳转到受控入口；内核完成受保护操作后恢复用户上下文。切换本身和可能发生的调度、缓存影响都会产生开销。

### 98. 线程上下文切换

系统保存程序计数器、栈指针、通用寄存器、状态寄存器以及调度和线程局部状态，再恢复另一线程。跨进程切换还可能改变地址空间映射，增加 TLB 与缓存代价。

### 99. 虚拟内存

虚拟内存为每个进程提供隔离、连续的地址视图，通过页表映射到物理页或磁盘。它支持保护、共享、按需加载和超出物理内存的地址空间，但页表、TLB miss 和缺页会产生开销。

### 100. 缺页中断

CPU 访问无有效映射的页时陷入内核。内核验证访问是否合法，找到空闲页或淘汰牺牲页，必要时从磁盘读入数据，更新页表与 TLB 后重启指令；非法访问则终止进程或发出异常。

### 101. 页面置换

常见策略有 FIFO、LRU、Clock/Second Chance 和近似 LRU。理论最优算法只用于比较；实际系统通常结合访问位、脏位、工作集和后台回写，避免严格 LRU 的高维护成本。

### 102. 进程间通信

方式包括管道、命名管道、消息队列、共享内存、信号、socket 和文件映射。共享内存吞吐高但需要额外同步；socket 跨机器统一；管道与消息队列接口简单但会复制或经过内核缓冲。

### 103. mutex、spinlock 与 semaphore

互斥锁竞争时让线程睡眠，适合临界区较长或竞争明显；自旋锁忙等，适合临界区极短且不能睡眠的情况；信号量维护许可计数，适合限制并发资源数量或线程间通知。选择需比较等待时长与上下文切换成本。

### 104. condition_variable

条件变量只负责等待和通知，不保存业务条件。等待必须在锁保护下循环检查谓词，以处理虚假唤醒和通知前条件已变化；wait 会原子地释放锁并休眠，醒来后重新加锁。

### 105. 原子操作与锁

原子保证单个或一组受支持操作不可分割，并可建立内存顺序；锁可保护任意复杂不变量，但可能阻塞和切换。无锁不等于无等待，也不自动更快，复杂结构还要处理 ABA 和内存回收。

### 106. 死锁四条件

互斥、占有且等待、不可抢占、循环等待同时成立才可能死锁。破坏其中任一条件即可预防，例如统一锁顺序破坏循环等待。

### 107. 死锁处理

预防可固定加锁顺序或一次获取全部资源；避免可在资源分配前判断安全状态；检测可构建等待图；解除可回滚、抢占或终止参与者。工程中还常用 try_lock、超时和缩小临界区。

### 108. 线程安全懒汉单例

C++11 的函数局部静态变量初始化由语言保证线程安全：`static T instance; return instance;`。若必须手工控制，可用 `std::call_once`；双重检查需要正确的原子和内存序，通常没有必要。

### 109. False Sharing

不同线程修改逻辑上无关、但位于同一 cache line 的数据时，缓存一致性协议会让该行在核心间反复失效。可通过拆分写热点、按 cache line 对齐或每线程累加后归并来缓解，并用性能工具验证。

### 110. 游戏线程同步等待

主线程、渲染线程或 RHI/GPU 彼此等待会直接拉长帧时间并形成气泡。应减少共享可变状态，用任务依赖、双缓冲、批处理和异步 IO，让同步点位于真正需要结果的位置。

## 网络与同步

### 111. TCP 与 UDP

TCP 面向连接，提供可靠、有序的字节流并包含流量与拥塞控制；UDP 面向数据报，不保证到达、顺序或唯一性，但保留消息边界且开销较低。实时游戏常在 UDP 上按业务需要选择可靠性，而账户、下载等可直接使用 TCP/HTTP。

### 112. 三次握手

三次握手让双方确认发送与接收能力，并同步双方初始序列号。两次无法让服务端确认客户端已收到自己的序列号，也更难排除历史重复连接请求造成的半开状态。

### 113. 四次挥手与 TIME_WAIT

TCP 两个方向独立关闭，一方 FIN 后另一方可能仍需发送数据，所以通常需要四段。主动关闭方保留 TIME_WAIT，以便重发最后 ACK，并让旧连接的延迟报文在网络中消失，避免污染同四元组的新连接。

### 114. TCP 粘包

TCP 是字节流，没有应用消息边界；一次 send 与一次 recv 不一一对应。应用层需使用定长消息、分隔符或“长度头+正文”协议，并处理半包、多包和恶意长度。

### 115. 可靠 UDP

为数据包增加连接/通道标识、序号、ACK 或选择确认、重传计时器、重复过滤和乱序缓冲；再加入滑动窗口、流量控制与拥塞控制。不同消息应支持可靠有序、可靠无序和不可靠等语义，避免所有实时数据都等待重传。

### 116. 停等、滑动窗口与选择重传

停等每发一包等待 ACK，简单但带宽时延积利用低。滑动窗口允许多个未确认包在途；Go-Back-N 丢包后重传该包及后续包，选择重传只补丢失包，效率更高但状态更复杂。

### 117. 状态同步与帧同步

状态同步由服务端运行逻辑并发送实体快照，适合复杂、非完全确定的世界，带宽较高但客户端容错好。帧同步广播玩家输入，所有端运行确定性模拟，带宽低且适合回放，但对确定性、作弊防护和掉线恢复要求高。

### 118. 服务端权威

服务端拥有最终游戏状态，客户端发送意图而不是结果。服务端验证位置、资源、冷却和权限后修改状态，客户端可预测表现但必须接受校正，从而限制伪造伤害、物品或移动。

### 119. 插值、外推、预测与校正

插值在延迟缓冲中的两个已知快照间平滑，稳定但增加显示延迟；外推按速度推测未来，延迟低但容易误差；本地预测立即模拟自身输入；服务端结果到达后校正，并重放未确认输入以保持手感。

### 120. 延迟补偿与回滚

服务端按时间保存关键碰撞体的位置、姿态和状态快照。收到带时间戳的射击请求后，把相关对象临时回溯到历史时刻验证，再恢复当前状态；必须限制回溯窗口并防止客户端伪造过旧时间。

### 121. 商店价格

客户端只能提交商品 ID、数量和请求 ID，价格、折扣、库存与余额由服务端数据决定。服务端还需保证操作原子性和幂等性，防止重放、并发购买与断线重试造成重复扣款或发货。

### 122. 拾取物竞争

服务端持有拾取物权威状态，在同一串行或原子事务中验证存在、距离与资格，成功加入背包后再标记已领取并销毁。后到请求看到状态已改变就失败，客户端预测拾取必须等待服务端确认。

### 123. 序列化、量化与增量同步

序列化只发送协议所需字段；量化把高精度浮点映射为足够范围的较少位数；增量同步只发送相对基线发生变化的数据。还可用位域、字典 ID、条件复制和兴趣管理减少带宽。

### 124. RPC 与状态复制

RPC 适合一次性命令或事件，如请求交互和播放短暂表现；状态复制适合需要长期保持且晚加入者也应看到的数据，如生命和门的开关。不要用 RPC 模拟持久状态，也不要为每个瞬时粒子创建长期复制变量。

### 125. 网络指标影响

延迟让输入反馈变慢；丢包导致可靠数据重传或不可靠数据缺失；抖动让包到达间隔不稳定，需要缓冲；带宽不足造成排队和丢弃。解决方案分别涉及预测、冗余/重传、抖动缓冲、压缩和优先级。

## 游戏数学与图形学

### 126. 点积

`dot(a,b)=|a||b|cosθ`。单位方向点积大于零表示夹角小于 90°，越接近 1 越同向；与 `cos(FOV/2)` 比较可做视锥角度筛选。计算角度前要归一化并钳制浮点误差。

### 127. 叉积

三维叉积结果垂直于两向量，方向遵循右手定则，长度等于平行四边形面积。二维可取叉积 z 分量判断左右转向，符号还取决于坐标系约定。

### 128. 点在三角形内

可用三条边与点形成的叉积符号是否一致；也可求重心坐标并检查三个权重都在 `[0,1]` 且和为 1；面积法比较三个子三角形面积和。要定义边界算内还是外并使用浮点容差。

### 129. 圆内均匀随机

角度取 `θ=2πU`，半径取 `r=R*sqrt(V)`，坐标为 `(r cosθ,r sinθ)`。若半径直接均匀取值，中心附近每单位面积概率会过高，因为圆环面积与半径成正比。

### 130. 锥体内散布

均匀立体角采样时，方位角均匀取 `[0,2π)`，`cosθ` 在 `[cosθmax,1]` 均匀取值，再以射击前向建立正交基转换到世界空间。直接均匀取 θ 会让锥轴附近分布不均。

### 131. 万向锁

欧拉角按顺序绕三个轴旋转，当其中两个旋转轴重合时会失去一个自由度。问题来自参数化和旋转顺序，不是浮点精度；四元数或旋转矩阵可避免插值路径上的这一退化。

### 132. 四元数与 Slerp

单位四元数用四个分量表示三维旋转，组合对应四元数乘法，避免欧拉角奇异。Slerp 沿四维单位球最短弧以恒定角速度插值；点积为负时通常翻转一端以选择短弧。

### 133. 坐标空间

模型矩阵把局部点变到世界，观察矩阵把世界变到相机空间，投影矩阵变到裁剪空间；列向量约定下常写 `P*V*M*p`。组合顺序取决于行列向量和引擎约定，回答必须先声明约定。

### 134. 渲染管线

CPU 组织场景并提交绘制，GPU 读取顶点，经顶点着色、可选细分/几何阶段、图元装配、裁剪和光栅化生成片元；片元着色后进行深度模板、混合并写入目标，最后呈现。现代 GPU 会并行和重排部分阶段，但逻辑顺序仍如此。

### 135. Shader 执行次数

顶点着色器按提交的顶点调用，索引与顶点缓存可能减少重复处理；片元着色器按覆盖样本和 early test 情况调用，与屏幕覆盖、过绘制、MSAA、裁剪和分支有关，不等于三角形或顶点数量。

### 136. 前向与延迟渲染

前向在绘制物体时直接计算光照，对透明和 MSAA 友好，但多光源会重复计算。延迟先写 GBuffer，再按光源做屏幕空间光照，适合大量动态光，但占带宽/显存，透明与多采样处理更复杂。

### 137. 透明与延迟光照

传统 GBuffer 每像素主要保存最前方表面，无法同时保存多个透明层及其混合顺序。透明物体通常在延迟不透明阶段后用前向路径按深度排序绘制，复杂交叠仍可能产生排序问题。

### 138. 深度、Early-Z 与透明

深度测试用已有深度拒绝被遮挡片元；Early-Z 在片元着色前执行以节省计算。Shader 写深度、discard、某些副作用或顺序要求会限制提前测试。透明通常读深度但不写深度，并按后到前混合。

### 139. MSAA、TAA 与 FXAA

MSAA 对覆盖率多采样，几何边缘质量好但成本随样本数增加；FXAA 是快速后处理边缘平滑，便宜但会模糊；TAA 利用运动向量和历史帧累积，稳定亚像素细节但可能拖影、鬼影和变软。

### 140. Shadow Map

先从光源视角渲染深度，再在相机渲染时把像素投影到光空间比较深度。有限分辨率和偏移会造成锯齿、阴影痤疮、Peter Panning；常用级联、PCF、合理 bias 和更高分辨率缓解。

### 141. 骨骼蒙皮

每个顶点按若干骨骼权重，把绑定姿态顶点乘以对应骨骼的当前变换与逆绑定矩阵，再加权求和。GPU skinning 在顶点着色器或计算着色器执行，减少 CPU 顶点处理但增加 GPU 带宽与骨骼数据上传。

### 142. Draw Call、Batch 与 Instancing

Draw Call 有 CPU 状态验证、命令构建和驱动提交成本。Batch 合并兼容几何以减少提交；Instancing 用一次绘制加每实例变换绘制相同网格/材质。合并受材质、状态、透明顺序和剔除粒度限制。

### 143. 空间结构选择

均匀网格适合密度较均匀且动态对象多；四叉树/八叉树适合分层空间但对象跨节点处理复杂；BVH 按对象包围盒分组，对不规则几何和射线查询常有效。选择取决于维度、动态更新、密度和查询类型。

### 144. CPU/GPU 同步

CPU 读取 GPU 结果、复用仍在使用的资源或命令队列耗尽时会等待；GPU 也可能因资源上传和依赖空闲。使用多缓冲、异步读回、批量上传、帧资源轮换和减少强制 flush，避免在帧中间等待结果。

### 145. 判断瓶颈线程

比较 Game、Draw/Render、GPU 帧时间：最大的阶段通常限制帧率。随后用 Unreal Insights 定位 CPU 事件、ProfileGPU 定位 GPU pass，并在固定场景验证；仅看 FPS 无法区分瓶颈，也要留意帧同步导致各线程时间互相抬高。
